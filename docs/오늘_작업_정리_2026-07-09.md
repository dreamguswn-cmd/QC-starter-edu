## 3. `/fault-lab`에 지표(metrics) 계측 추가

### 대상 파일

- `app/main.py`

### 문제

`/fault-lab`은 장애(지연·500·타임아웃)를 재현하는 실습용 엔드포인트인데, 정작 `agent_request_total` 등 어떤 Prometheus 지표도 증가시키지 않아 아무리 호출해도 Grafana에 흔적이 남지 않았다. `/health`도 같은 문제였다.

### 조치

`fault_lab()` 핸들러에 `/ask`와 동일한 방식으로 계측을 추가했다.

- 성공(`normal`/`delay`) → `agent_request_total{mode="fault-lab",status="success"}`, `agent_success_total`, `agent_response_seconds`
- 실패(`error500`/`timeout`/잘못된 scenario) → `agent_request_total{mode="fault-lab",status="error"}`, `agent_error_total`, `agent_response_seconds`

### 검증

- 도커 이미지 재빌드 후 `error500` 3회, `delay=1초` 1회 호출 → `agent_error_total{mode="fault-lab"}=3`, 응답시간 히스토그램의 1~2초 구간에 정확히 반영됨을 실측 확인
- Prometheus PromQL로 동일 값 재확인
- `pytest tests/test_fault_lab.py` 4건 전부 PASSED (회귀 없음)

## 5. Grafana 대시보드 개선

### 대상 파일

- `monitoring/grafana_dashboard.json`
- `monitoring/prometheus.yml`
- `docker-compose.yml`

### 문제 1 — 그래프가 다양하게 안 나옴

모든 패널이 `sum(...)`으로 전체를 하나로 합산해서, mode(rule/api/fault-lab)나 status(success/error)가 달라도 선이 하나로만 보였다.

**조치**: `sum by (mode) (...)`, `sum by (status) (...)`로 분리하고, 신규 패널 2개를 추가했다.

- "모드별 요청 수 추이" — rule/api/fault-lab 별 선
- "fault-lab 성공/오류 비율 추이" — success/error 별 선

### 문제 2 — 1초 단위로 안 보임

- `prometheus.yml`의 `scrape_interval`을 5s → **1s**로 변경
- Grafana는 기본적으로 5초 미만 새로고침을 막기 때문에(`min_refresh_interval`), `docker-compose.yml`에 `GF_DASHBOARDS_MIN_REFRESH_INTERVAL=1s` 추가
- `grafana_dashboard.json`의 `refresh`를 `5s` → `1s`
- 실측: `agent_service_status`를 `step=1s`로 range 조회해 샘플 간격이 정확히 1.0초임을 확인

### 문제 3 — Grafana 열기 버튼을 눌러도 로그인 화면만 보임

- `docker-compose.yml`에 `GF_AUTH_ANONYMOUS_ENABLED=true`, `GF_AUTH_ANONYMOUS_ORG_ROLE=Viewer` 추가
- Grafana HTTP API로 Prometheus 데이터소스와 대시보드 JSON을 **직접 등록**(수동 Import 과정 생략)
- 익명 상태로 대시보드 URL을 호출해 200 응답과 정상 콘텐츠(제목·패널 10개·refresh=1s)를 실측 확인

## 6. 트래픽 생성 스크립트 작성 및 실행

### 대상 파일

- `load_traffic.ps1` (신규)

### 작업 내용

Grafana 그래프에 다양한 데이터를 채우기 위해 1000건짜리 PowerShell 트래픽 스크립트를 작성했다.

| 비중 | 종류 |
|---|---|
| 60% | `/ask` (10가지 질문 랜덤, rule 모드) |
| 20% | `fault-lab normal` |
| 10% | `fault-lab delay` (0.1~1.5초 랜덤) |
| 5% | `fault-lab error500` |
| 5% | `fault-lab timeout` (0.5~2초 랜덤) |

요청 사이에 50~250ms 랜덤 대기를 둬서 순간 스파이크가 아니라 시간에 걸쳐 퍼지도록 했다.

### 결과

1000건 실행 완료 후 Prometheus에서 합계 1000 확인. 분포는 `rule` 596건, `fault-lab` 성공 312건, 오류 92건.

이어서 `/pipeline/run`을 8회(1초 간격) 호출해 `quality_eval_pass_total`/`quality_eval_fail_total`(PASS율 패널용 지표)도 채웠다 — PASS율 70%로 정상 조회 확인.

## 8. 오늘 작업의 포트폴리오 의미

어제가 "기능을 코드로 구현하고 문서화"하는 단계였다면, 오늘은 그 기능들이 **실제 운영 환경에서 관찰 가능한 신호로 이어지는지**를 직접 눈으로 확인하는 단계였다. 특히:

- 지표가 없으면 아무리 장애를 재현해도 모니터링에 보이지 않는다는 점(계측의 중요성)을 코드 수정으로 직접 해결
- Windows 환경 특유의 포트 바인딩 우선순위(구체 주소 vs 와일드카드) 문제를 실측으로 원인 규명
- PromQL의 `sum(...)` vs `sum by (label) (...)` 차이가 "데이터가 다양하게 안 보인다"는 체감 문제의 실제 원인이었음을 확인
- Grafana의 `min_refresh_interval` 같은 숨은 제약을 찾아서 해결

이 과정 전체가 "그래프에 아무것도 안 보인다" → "왜 안 보이는지 계층별로(앱 지표 → Prometheus 수집 → Grafana 쿼리/설정) 원인을 좁혀나가는" QA·운영 디버깅 사례로 정리된다.
