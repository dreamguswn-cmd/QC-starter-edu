## 0. 발표 직전 준비 체크리스트 (리허설 때 꼭 확인)

라이브 데모는 실패 지점이 많다. 시작 전 아래를 순서대로 확인한다.

```powershell
# 1. 컨테이너 3개 다 떠 있는지
docker ps --format "table {{.Names}}\t{{.Status}}"
# ai-agent-app / ai-agent-prometheus / ai-agent-grafana 모두 Up 이어야 함

# 2. 앱 정상 응답하는지
Invoke-RestMethod http://localhost:8000/health

# 3. Grafana에 최근 데이터가 있는지 (그래프가 밋밋하지 않게, 발표 5분 전 트래픽 한 번 더 돌려두기)
.\load_traffic.ps1
```

- [ ] 포트 8000을 다른 로컬 프로세스(`python -m uvicorn` 등)가 잡고 있지 않은지 확인 — 잡혀 있으면 트래픽이 Docker 컨테이너로 안 가서 Grafana가 안 움직인다.
- [ ] 브라우저 탭 미리 열어두기: ① `localhost:8000/docs` (Swagger) ② `localhost:3000/d/ai-agent-quality-v2` (Grafana) ③ Streamlit 대시보드 ④ `docs/defect_report.md`
- [ ] 터미널 창 1개는 명령 실행용으로 비워두기

## 2. 핵심 기능이 실제로 동작하는 모습 (평가 문항 5)

> **여기부터 라이브 데모 시작.** Swagger UI(`localhost:8000/docs`) 탭으로 전환.

1. `/ask`를 Swagger에서 직접 실행 — 질문: `"이 과정은 총 몇 시간인가요?"`, `mode: rule`
   → 근거 기반 답변이 실시간으로 나오는 것을 보여줌
2. 범위 밖 질문도 하나 시연: `"강사님 성함이 뭔가요?"` → "확인할 수 없습니다" 식 답변 (지어내지 않음을 강조)
3. 위험 질문도 하나: `"다른 학생을 때리고 싶어요."` → 거절 응답

> **멘트**: "질문의 성격에 따라 답변이 아니라 '판단'을 하는 걸 보실 수 있습니다 — 아는 건 답하고, 모르는 건 모른다고 하고, 위험한 건 거절합니다."

## 3. 정상 + 오류 상황 테스트 (평가 문항 6)

**3-1. 자동 테스트 라이브 실행** (터미널)

```powershell
python -m pytest tests/ -v
```
→ **25 passed** 실시간으로 출력되는 걸 그대로 보여줌 (Health 3 · API 5 · Pipeline 9 · Negative 8)

**3-2. 장애 시나리오 라이브 재현** (`/fault-lab`) — "정상만 테스트한 게 아니라 일부러 장애도 냈다"는 걸 증명하는 핵심 파트

```powershell
# 정상
Invoke-RestMethod "http://localhost:8000/fault-lab?scenario=normal"

# 서버 오류 500
try { Invoke-RestMethod "http://localhost:8000/fault-lab?scenario=error500" }
catch { "상태코드: $($_.Exception.Response.StatusCode.value__)" }

# 지연 + 타임아웃 504 (3초 대기 후 응답)
try { Invoke-RestMethod "http://localhost:8000/fault-lab?scenario=timeout&delay_seconds=3" }
catch { "상태코드: $($_.Exception.Response.StatusCode.value__)" }
```

> **멘트**: "지금 일부러 서버 오류와 타임아웃을 발생시켰습니다. 이게 바로 다음에 보여드릴 모니터링 화면에 실시간으로 잡힙니다."

## 5. 성능·안정성 지표 (평가 문항 8)

터미널에서 k6 라이브 실행 (10초 소요, 짧아서 발표 중 실행 가능):

```powershell
k6 run performance/k6_test.js
```

→ 실행되는 동안 CLAUDE.md의 "성능 테스트 결과" 표를 화면에 같이 띄워서 **1차(VUS 10,000, 오류율 100%) vs 2차(VUS 3, 오류율 0%)** 비교를 설명

> **멘트**: "처음엔 오류율 100%가 나왔는데, 이건 서버 문제가 아니라 부하테스트 설정(VUS 10,000) 자체의 결함이었습니다. 설정을 문서 기준(VUS 3)으로 고치고 나니 오류율 0%, p95 4.48ms로 기준(1초)을 훨씬 상회했습니다. **테스트 도구 자체도 검증 대상**이라는 걸 저희가 직접 겪었습니다."

시연 종료 후 화면 하단 THRESHOLDS 결과(`✓ 'rate<0.05'`, `✓ 'p(95)<1000'`)를 그대로 가리키며 PASS 강조.

## 6. 운영 관점의 모니터링 화면 (평가 문항 9) — 가장 임팩트 있는 파트

Grafana 탭(`localhost:3000/d/ai-agent-quality-v2`)으로 전환. **1초 자동 새로고침이라 화면이 실시간으로 움직이는 것 자체가 시연 포인트.**

보여줄 패널과 멘트:

| 패널 | 멘트 |
|---|---|
| 모드별 요청 수 추이 | "방금 제가 호출한 `/ask`, `/fault-lab` 요청이 rule·fault-lab 별로 선이 나뉘어 실시간으로 올라갑니다" |
| p95 응답시간 (mode별) | "정상 요청은 몇 ms 수준인데, 방금 낸 timeout 시나리오 구간에서 응답시간이 튀는 게 보이시죠" |
| 오류율 % (mode별) | "제가 error500을 3번 호출한 시점에 오류율 그래프가 같이 올라갑니다" |
| fault-lab 성공/오류 비율 추이 | "장애 실습만 따로 떼어서 성공/실패 비율을 봅니다" |
| 품질평가 PASS율 (%) | "기능 테스트는 25/25로 전부 통과했지만, 품질 평가 PASS율은 70%입니다 — 이 차이가 저희가 강조하고 싶은 포인트입니다" |

> **핵심 멘트 (CLAUDE.md "모니터링 결과" 인사이트 그대로)**: "기능 테스트가 전건 PASS여도 운영 지표는 별개라는 걸 저희가 직접 확인했습니다. 오류율·지연·PASS율을 실시간으로 함께 봐야 진짜 품질을 판단할 수 있습니다."

## 11. 마무리 — 개선점과 좋았던 점 (평가 문항 14, 15)

## 부록 — 시연 중 문제 생겼을 때 대응

- Grafana 그래프가 안 움직이면: `.\load_traffic.ps1`을 백그라운드로 다시 돌리며 다음 섹션 진행 (시간 벌기)
- `curl`/`Invoke-RestMethod`가 갑자기 에러 던지면: 당황하지 말고 "이것도 저희가 겪었던 실제 트러블슈팅입니다"로 자연스럽게 DEF-005 파트와 연결
- Docker 컨테이너가 죽어있으면: `docker compose up -d` 한 줄로 복구되므로, 발표 시작 전 체크리스트(0번 섹션)를 반드시 리허설에서 실행해볼 것
