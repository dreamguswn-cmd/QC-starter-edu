## 원인: `performance/k6_test.js`의 `vus` 설정 오류

```js
export const options = {
  vus: 10000,        // ← 주석("문서 기준: vus=3, duration=10s")과 실행_가이드.md의
  duration: '30s',   //    "0→30 VU 램프업" 설계 의도와 전혀 다른 값
  ...
};
```

- 가상 사용자(VU) **10,000명**이 동시에 로컬 단일 프로세스 `uvicorn --reload` 서버의 `/health`에 30초간 요청을 퍼부음
- 서버가 이 부하를 감당하지 못해 TCP 연결 단계에서부터 대량 실패 → 응답 자체를 받지 못함(`data_received: 0 byte`, `data_sent: 0 byte`)
- 실제 에러 로그(반복 발생):
  ```
  GoError: the body is null so we can't transform it to JSON - this likely was
  because of a request error getting the response
    at status is ok (k6_test.js:42:41)
  ```
- 즉, 이번 테스트는 **API 성능을 측정한 것이 아니라, 잘못된 설정으로 로컬 서버에 사실상 DoS성 부하를 가한 것**에 가깝습니다.

## 부가 발견: `MODE` 환경변수 미사용

`$env:MODE="api"`를 설정해도 `performance/k6_test.js`에는 `__ENV.MODE`를 참조하는 코드가 없어 아무 효과가 없습니다. 스크립트는 항상 `/health`만 호출하며, `mode=api`로 `/ask`를 호출하는 시나리오는 현재 구현되어 있지 않습니다.

## 부가 발견: 1차 부하로 인한 포트 8000 좀비 소켓

1차(vus=10000) 실행 이후, 해당 부하를 처리하던 uvicorn 프로세스가 강제 종료된 뒤에도 Windows 커널 소켓 테이블에 `127.0.0.1:8000` Listen 항목이 남아 신규 연결을 계속 거부(`Connection refused`)했습니다. `Get-Process`로는 해당 PID가 이미 존재하지 않는데도 `Get-NetTCPConnection`에는 Listen 상태로 남아있는 상태였습니다. `taskkill`로도 해결되지 않아, 임시로 `--port 8001`로 우회해 재실행했습니다. 8000번 포트는 시간이 지나면(OS 소켓 정리 주기 이후) 자연스럽게 해제될 가능성이 높으며, 필요 시 PC 재부팅으로 즉시 해제할 수 있습니다.
