# 이 프로젝트는 무엇인가요? (초보자를 위한 설명서)

## 한 줄 요약
"교육과정 안내 챗봇"을 만들고, 그 챗봇이 **제대로 답하는지 자동으로 검사·감시**하는 시스템입니다.

---

## 1. 왜 만들었나요?

챗봇을 하나 만드는 건 어렵지 않습니다. 어려운 건 그다음입니다.

- 챗봇이 **틀린 답**을 하면 어떻게 알아채나요?
- 챗봇이 **모르는 질문**에 아무 말이나 지어내면 어떻게 막나요?
- 챗봇이 **위험한 요청**(예: "친구를 때리고 싶어요")에 어떻게 반응해야 하나요?
- 서버가 **느려지거나 죽으면** 누가 먼저 알아채나요?
- 갑자기 **사용자가 몰리면** 서버가 버틸 수 있나요?

이 프로젝트는 "챗봇을 만드는 것"이 아니라, **"챗봇의 품질을 개발 → 테스트 → 배포 → 운영 감시까지 전 과정에서 관리하는 것"**을 보여주는 포트폴리오입니다.

---

## 2. 전체 그림 (한눈에 보기)

```
[사용자 질문]
      ↓
[FastAPI 서버] ──→ [챗봇이 답변 생성]
      ↓
[품질 검사기] ──→ 규칙 검사(키워드 맞나?) + AI 심사관(의미 맞나?)
      ↓
[리포트 생성] ──→ 결과를 표/그래프/문서로 저장
      ↓
[대시보드 & 모니터링] ──→ 사람이 한눈에 확인
```

즉 "챗봇 1개"가 아니라, 챗봇을 둘러싼 **품질 관리 파이프라인 전체**가 이 프로젝트입니다.

---

## 3. 폴더 구조 — 각각 무슨 역할인가요?

```
app/          챗봇 서버 본체 (FastAPI)
quality/      챗봇 답변이 맞는지 검사하는 로직
tests/        "정말 잘 작동하는지" 자동으로 확인하는 테스트 코드
dashboard/    검사 결과를 그래프로 보여주는 화면 (Streamlit)
monitoring/   서버가 살아있는지, 느려지지 않았는지 감시 (Prometheus·Grafana)
performance/  사용자가 몰릴 때도 버티는지 확인하는 부하 테스트 (k6)
docs/         테스트 계획서, 결함 보고서, 최종 보고서 등 문서
```

### app/ (챗봇 서버)
| 파일 | 역할 |
|---|---|
| `main.py` | 서버의 정문. 모든 요청(`/ask`, `/health` 등)이 여기로 들어옵니다 |
| `service_agent.py` | 실제로 질문에 답하는 로직 (키워드 기반 또는 OpenAI 연동) |
| `knowledge_base.py` | 챗봇이 알고 있는 "정답 정보"(교육시간, 출결 규정 등) |
| `judge_agent.py` | AI가 스스로 "이 답변이 잘 된 답변인지" 채점하는 심사관 역할 |
| `fault_lab.py` | 일부러 장애(지연·오류·타임아웃)를 재현해서 대응 훈련을 하는 실습용 코드 |
| `metrics.py` | "몇 번 요청됐는지", "몇 번 실패했는지" 같은 숫자를 기록 |

### quality/ (품질 검사)
| 파일 | 역할 |
|---|---|
| `rule_validator.py` | "답변에 기대하는 단어가 들어있는가?" 같은 단순 규칙 검사 |
| `quality_pipeline.py` | 여러 질문을 한 번에 돌려서 전체 성적표를 만드는 로직 |
| `report_generator.py` | 검사 결과를 JSON·CSV·마크다운 문서로 저장 |
| `test_cases.json` | 미리 준비된 "질문 + 기대하는 답" 목록 |

### tests/ (자동 테스트)
사람이 매번 손으로 확인하지 않아도, `pytest` 명령 한 줄로 **서버가 여전히 잘 작동하는지** 자동으로 검증합니다. 코드를 수정한 뒤 "혹시 뭔가 망가지지 않았나?"를 몇 초 만에 확인할 수 있습니다.

### dashboard/ (대시보드)
검사 결과(몇 개 통과했는지, 어떤 게 실패했는지)를 웹 화면의 그래프와 표로 보여줍니다.

### monitoring/ (모니터링)
서버가 지금 살아있는지, 응답이 느려지고 있는지를 실시간 그래프로 감시합니다. 병원의 심전도 모니터와 비슷한 역할입니다.

### performance/ (부하 테스트)
"동시에 30명이 몰려도 서버가 버티는가?"를 미리 시뮬레이션해서 확인합니다.

---

## 4. 챗봇은 어떻게 답을 만드나요?

두 가지 모드가 있습니다.

1. **rule 모드** (기본, API 키 필요 없음)
   - 질문에 특정 키워드가 있는지 확인해서 정해진 답을 돌려줍니다.
   - 예: "총 몇 시간인가요?" → "교육시간" 키워드 발견 → 미리 정해둔 시간 안내
   - 위험한 단어(때리, 협박 등)가 있으면 무조건 거절합니다.
   - 모르는 질문이면 "확인할 수 없습니다"라고 솔직하게 답합니다 (아무 말이나 지어내지 않음).

2. **api 모드** (OpenAI API 키 필요)
   - 실제 AI(GPT)에게 "이 정보만 근거로 답해"라고 시켜서 답을 만듭니다.

---

## 5. 답변이 "잘 됐는지"는 누가 판단하나요?

두 단계로 검사합니다.

1. **규칙 검사** (rule_validator.py): 답변에 기대하는 단어가 들어있는지 기계적으로 확인
2. **AI 심사관** (judge_agent.py): 정확성 · 근거성 · 유용성 · 안전성 4가지 기준으로 점수(1~5점)를 매김

이 둘을 합쳐서 최종 **PASS / FAIL**을 결정합니다.

---

## 6. 실제로 실행해보려면?

```bash
# 1. 필요한 패키지 설치
pip install -r requirements.txt

# 2. 서버 실행
uvicorn app.main:app --reload
# → http://localhost:8000/docs 접속하면 직접 테스트 가능

# 3. 자동 테스트 실행 (서버 안 켜도 됨)
pytest tests/ -v

# 4. 품질 검사 전체 실행 + 보고서 생성
python -m quality.quality_pipeline

# 5. 결과를 그래프로 보기
streamlit run dashboard/streamlit_app.py
```

더 자세한 단계별 설명은 [docs/실행_가이드.md](docs/실행_가이드.md) 문서를 참고하세요.

## 7. "장애 실습(fault_lab)"은 왜 있나요?

실제 서비스는 언젠가 반드시 문제가 생깁니다. 응답이 느려지거나, 서버 내부 오류가 나거나, 타임아웃이 발생합니다.

`fault_lab.py`와 `/fault-lab` API는 이런 상황을 **일부러 재현**해서, "장애가 났을 때 어떻게 감지하고 기록하고 보고하는지"를 연습하기 위한 코드입니다. 실제 버그가 아니라 **훈련용 시뮬레이터**입니다.

```
/fault-lab?scenario=normal    → 정상 응답
/fault-lab?scenario=delay     → 일부러 응답 지연
/fault-lab?scenario=error500  → 일부러 서버 오류
/fault-lab?scenario=timeout   → 일부러 타임아웃
```

이렇게 발생시킨 장애 상황을 `docs/defect_report.md` 같은 문서에 "결함 보고서" 형식으로 기록하는 연습도 함께 합니다.

## 8. 한눈에 보는 용어 정리

| 용어 | 쉬운 설명 |
|---|---|
| FastAPI | 파이썬으로 웹 서버(API)를 만드는 도구 |
| pytest | 코드가 잘 작동하는지 자동으로 확인하는 테스트 도구 |
| Judge Agent | AI가 다른 AI의 답변을 채점하는 역할 |
| Prometheus | 서버 상태(요청 수, 응답 시간 등) 숫자를 수집하는 도구 |
| Grafana | 수집된 숫자를 그래프로 예쁘게 보여주는 도구 |
| k6 | 동시에 여러 명이 접속했을 때를 흉내 내는 부하 테스트 도구 |
| Streamlit | 파이썬 코드만으로 웹 대시보드를 만드는 도구 |
| Docker | 여러 프로그램(서버+모니터링 등)을 한 번에 묶어서 실행하는 도구 |
